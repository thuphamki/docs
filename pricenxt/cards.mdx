## Card fundamentals

Cards are the atomic insight units inside every board column. They wrap data fetching, formatting, and visualization logic behind a consistent API so users can mix and match analytics without writing queries.

Each card instance stores two things:

1. **Title** – human-readable label shown on the card chrome.
2. **Metadata** – type-specific configuration captured via the card configurator (variant IDs, comparison lists, gauge thresholds, etc.).

```ts
export interface CardFormData<T extends CardMetadata = CardMetadata> {
  readonly title: string;
  readonly metadata: T;
}
```

## Lifecycle

1. Users open the card configurator (`ConfiguratorStep: select → configure`).
2. Selected card type provides a default metadata payload from the registry.
3. Form validation (`CardFormValidation`) ensures required fields and variants are present.
4. Upon save, the card is persisted under the active column and rendered through the card registry.

<Tip>Metadata objects are read-only by design. Always clone before mutating local state inside forms.</Tip>

## Card types

| Type | Best for | Key metadata |
| --- | --- | --- |
| `price` | Single variant spot or monthly price tracking | `variantId`, `deltaPeriod`, `showForecast`, `chartStyle` |
| `delta` | Highlighting period-over-period deltas | `variantId`, `deltaPeriod`, `showForecast` |
| `list` | Monitoring multiple variants in a single list | `variantIds[]`, `deltaPeriod`, optional highlight color |
| `compare` | Side-by-side comparison of up to four variants | `variantIds[]`, `deltaPeriod` |
| `gauge` | Threshold-driven KPI view (e.g., utilization) | `variantId`, `gauge{min,max,t1}`, `chartStyle` |
| `spread` | Tracking spreads between two variants | `variantIds[]`, `deltaPeriod`, `calculationMode` |

All available IDs live in `CARD_TYPE_OPTIONS`, ensuring filters, selectors, and Supabase enums stay aligned.

```ts
export const CARD_TYPE_OPTIONS = [
  { id: "price", labelKey: "board.filters.card-types.price" },
  // ...
];
```

## Categories & status badges

- **Categories** (`CardCategory`) cluster cards when browsing: commodity, capacity, arbitrage, currency, plastixx, news.
- **Status** (`CardStatus`) flags experimental types (e.g., `coming-soon`). The configurator displays these as Untitled UI badges.

<Callout icon="wand-magic">Use category + status metadata to drive feature-flag rollouts without branching the registry.</Callout>

## Display vs. preview components

- `CardPreviewProps` render lightweight previews while configuring a board.
- `CardDisplayProps` power the live dashboard. They receive optional `onEdit` handlers so the pencil icon can relaunch the configurator with existing metadata.

Keeping these props aligned ensures cards render identically across creation, duplication, and drag-and-drop flows.
